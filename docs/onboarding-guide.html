1. まずは「全体像」をつかもう

この仕組みでは 3つのサービス を使い分けます。最初に「誰が何の役割か」を知っておくと、あとの手順がぐっと分かりやすくなります。難しく考えず、下の3人組をイメージしてください。

🤖
Claude Code

役割：作る人（AI）
あなたの「こういうサイトが欲しい」という言葉を聞いて、代わりにコード（サイトの中身）を書いてくれるAIアシスタントです。

🐙
GitHub

役割：保管庫＆共有場所
作ったデータをクラウド上に保存します。変更の履歴が残り、みんなで共有できる「共同の倉庫」です。

役割：公開する場所
レンタルサーバーです。ここに置かれたデータがインターネット上に公開され、世界中の人が見られるようになります。

💡 なぜ、わざわざ3つも使うの？

「Claude Code だけで作って終わり」にできない理由は、役割が違うからです。Claude Code は“作る”のは得意ですが、それ単体では保存も公開もできません。GitHub に置くことで全員で共有でき、変更の履歴が残るので「間違えても前の状態に戻せる」ようになります。そして heteml に置いて初めて、サイトがインターネットに公開されます。3つで役割分担しているわけです。

🔑 いちばん大事な一言： あなたは基本的に Claude Code に「言葉でお願いする」だけ。GitHub への保存や heteml への公開は、ほとんど自動で進みます。仕組みは次ページ以降でゆっくり説明します。

2. 先に「用語」だけ（最低限でOK）

専門用語が出てくると不安になりますが、全部を覚える必要はありません。「だいたいこんな意味」とイメージできれば十分です。困ったらこのページに戻ってきてください。

用語やさしい説明（たとえ）
GitHub
(ギットハブ)		ネット上にある「共有フォルダ＋セーブ履歴」。みんなのデータが集まる倉庫。
リポジトリ
(repo / レポ)		プロジェクト1個分のフォルダのこと。今回のチームの倉庫は 「atest」 という名前です。
コミット
(commit)		ゲームの「セーブ」。ここまでの状態を1区切りとして記録すること。
プッシュ
(push)		セーブした内容をGitHub（クラウド）に送ること。これで他の人にも共有されます。
ブランチ
(branch)		作業用の「枝分かれ」。本番と分けて、自分の作業を安全に進める場所。
main
(メイン)		本番のブランチ。ここに変更が入ると heteml に公開されます（＝公開スイッチ）。
プルリクエスト
(PR)		「この変更を本番(main)に入れていいですか？」という申請書。
マージ
(merge)		その申請を本番(main)に取り込むこと＝公開の確定。
Claude CodeAIにお願いしてサイトを作ってもらうツール。チャットのように言葉で指示します。
heteml
(ヘテムル)		レンタルサーバー。サイトの公開先（インターネットに見える場所）。
デプロイ
(deploy)		作ったものをサーバーに配置して公開する作業のこと。
GitHub ActionsGitHubの「自動係（ロボット）」。mainが更新されたら、自動でhetemlへ送ってくれます。
FTP / FTPSサーバーにファイルを送る通信方法。手動アップのときに使います。
.gitignore「GitHubに載せない物リスト」。パスワード等の大事な情報を載せないための設定。
🧭 覚え方のコツ： 「コミット＝セーブ」「プッシュ＝クラウドに送る」「main＝公開スイッチ」。この3つだけ頭に入れておけば、まずは大丈夫です。

① github.com にアクセスして、無料アカウントを作ります（メールアドレス・パスワード・ユーザー名を登録）。

📷ここに「GitHub 新規登録画面」または「自分のユーザー名が分かる画面」のスクリーンショットを貼ってください
💡 なぜ、ユーザー名を共有するの？

💡 なぜ、招待が「2通」あるの？

スタートアップガイド作業場所のルール（フォルダ構成）

4. 作業場所のルール（自分のフォルダを作る）

倉庫「atest」は 複数人で共有しています。お互いの作業がぶつからないよう、「自分専用のフォルダ」を作って、その中だけで作業するのが基本ルールです。

📁 倉庫「atest」の中身のイメージ

atest/                        ← 共有リポジトリ（みんなの倉庫）
├─ api_banto_san/             ← 既にある人のプロジェクト（さわらない）
├─ ◯◯◯（あなたの名前）/        ← 【ここを新しく作る】あなたの作業場所
│    ├─ index.html              例）トップページ
│    ├─ style.css               例）デザイン
│    └─ images/                 例）画像など
├─ README.md                  ← 説明書（公開対象外）
└─ .github/                    ← 自動公開の設定。絶対にさわらない！

Claude Code に作業を頼むときは、「私のフォルダ（◯◯◯）の中に〜を作って」のように、自分のフォルダ名を必ず伝えるのがコツです。

💡 なぜ、自分のフォルダを作るの？

1つの倉庫を大勢で使うため、みんなが同じ場所を同時にいじると「衝突（コンフリクト）」や「上書き事故」が起きやすいからです。自分の区画（フォルダ）の中で作業すれば、他の人の作業とぶつからず、安全で、整理整頓もしやすいです。

🚫 GitHub（倉庫）の中身を消さないでください
倉庫の中には他の人が作った大切なデータが入っています。それを消すと…
　・他のメンバー全員に影響します（その人のサイトが消える）。
　・しかも main に反映されると heteml（本番サイト）からも消えてしまい、公開中のサイトが壊れることがあります。
履歴から元に戻すことは可能ですが、手間がかかり、その間サイトが見られなくなることもあります。
原則：「自分のフォルダの中」だけを編集する。他人のフォルダ・.github・README などはさわらない。不要そうに見えても、消す前に必ず相談してください。

自動で上がるもの／手動アップが必要なもの

ほとんどのファイルは自動で公開されますが、一部だけ「手動でアップ」が必要です。これが「たまに自分でファイルのアップが必要だよ」の正体です。

✅ 自動で公開されるもの✋ 手動アップが必要なもの
ふつうのサイトのファイル全般
HTML・CSS・JavaScript・PHP・画像 など、GitHubに保存した（コミットした）ファイル。
→ 何もしなくてOK。mainに入れば自動で公開。 		「GitHubに載せない物」＝.gitignore 対象
・config.local.php（パスワード等の設定ファイル）
・uploads/（アップロードされたファイル）
・ログやデータベース（*.log / *.db）など
→ GitHubに載らない＝自動で上がらない。必要なら heteml に直接アップ。
💡 なぜ、わざわざ手動アップが必要なものがあるの？

パスワードなどの「秘密の情報」は、安全のためにわざとGitHubに置いていないからです（誰でも見られると危険なため）。GitHubに無いものは自動で公開されないので、その分はサーバー（heteml）に直接置く必要があります。

📤 手動アップのやり方（ざっくり）

① heteml の管理画面にログイン → ② ファイルマネージャーやFTPで、対象ファイルを /web/atest/… の該当フォルダにアップロード。
ログイン情報（ID／パスワード）は担当者から共有されます：【要記入：共有方法／保管場所】

🔒 ID・パスワードは社外秘です。チャットやメールで外部に貼らない／公開リポジトリに書かない／人に教えない。万一の漏えいはサイト全体の事故につながります。

スタートアップガイド守ってほしいこと・FAQ・まとめ

7. 守ってほしいこと（ルールまとめ）

ルールなぜ？（理由）
① 自分のフォルダの中だけで作業する他の人の作業とぶつかる「衝突・上書き事故」を防ぐため。
② GitHub（特に他人のフォルダ）を消さない全員に影響し、本番サイト(heteml)まで壊れる恐れがあるため。
③ .github や設定ファイルはさわらない自動公開の仕組みが壊れ、誰も公開できなくなるため。
④ パスワード類を外部に貼らない／Gitに入れない情報漏えい＝サイト乗っ取り等の重大事故につながるため。
⑤ 迷ったら消さず・上書きせず、まず相談する「取り返しがつかない操作」を未然に防ぐため。

8. 困ったとき・よくある質問（FAQ）

こんなときどうする
招待メールが届かない迷惑メールを確認 → 無ければ担当者へ。リンクは期限切れの場合あり。
GitHub接続をミスした／間違えて押したもう一度「接続（Authorize）」をやり直せば直ります。慌てなくてOK。
自分のサイトのURLが分からないhttps://【要記入URL】/atest/自分のフォルダ/。正確なURLは担当者へ。
heteml に反映されない①mainに反映済みか確認 ②公開まで数分待つ ③手動アップが必要なファイルでないか確認。
何か消してしまったかも…すぐ担当者へ連絡。履歴から戻せる可能性が高いので、自分で慌てて操作しない。
用語が分からなくなった本資料 P.3「用語集」に戻って確認。全部覚えなくて大丈夫。
🙋 相談先：【要記入：担当者名・チャットルーム・連絡方法など】
「こんな初歩的なこと聞いていいの？」と思うことでも大歓迎です。事故を防ぐほうが大事なので、遠慮なく聞いてください。
